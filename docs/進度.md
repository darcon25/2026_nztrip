- **⚠️ 部署方式很特殊，改程式碼後不能只靠 `git push` 自動部署**：這個專案是 Vite 前端＋自訂 Express 後端混在一起，Zeabur 只要偵測到 Vite 就會強制當純靜態網站部署（環境變數、Dockerfile、改資料夾名稱都繞不過去，詳細除錯過程見 `docs/Zeabur部署教學.md`）。唯一可行解法：**本機把 Docker image 建好、推到 `ghcr.io/darcon25/2026_nztrip`，Zeabur 服務來源設成「Docker 映像檔」直接拉現成映像檔**。
- **每次要上線新版本，跑這幾行**（完整說明、常見問題排查見 `docs/Zeabur部署教學.md`）：
  ```bash
  cd ~/Projects/2026_nztrip  # 或對應的 worktree
  gh auth token | docker login ghcr.io -u darcon25 --password-stdin
  docker build --platform linux/amd64 --provenance=false --sbom=false \
    -t ghcr.io/darcon25/2026_nztrip:latest .
  docker push ghcr.io/darcon25/2026_nztrip:latest
  ```
  推上去後，去 Zeabur 服務頁面點「更新映像」觸發重新部署。**`--platform linux/amd64` 跟 `--provenance=false --sbom=false` 兩組參數缺一不可**（本機是 Apple Silicon，不指定平台會建出跟伺服器不合的架構；新版 Docker 預設的中繼資料格式 Zeabur 也不相容）。
- 舊的 Vercel 網址（`2026nztrip.vercel.app`）已棄用，後端從沒在上面部署成功過。

## 重要決策（別再重議）
- 家庭身分**用「名稱」**，不用 id 欄。
- 使用者產生的可變資料（分帳、留言、照片、煮飯菜色）一律存**網站後端 SQLite**（`comments.db`），不寫回 Google Sheet；Sheet 只當唯讀來源。
- 行程／住宿**以 Google Sheet 為準**；程式讀 Sheet 自動組版面，不寫死資料。
- 「誰住哪個房間」的房間分配功能，**使用者明確決定先不做**，Sheet 也沒有對應欄位。若日後要做，資料架構可參考自煮分工的 localStorage 身分 + 後端表模式。
- 快捷菜色按鈕清單（火鍋/白飯/烤肉/炒青菜/湯）是**暫定值**，等使用者提供實際常見菜色再換（`CookCalendar.tsx` 的 `QUICK_DISHES`）。

## QA 中修過的坑（避免重踩）
- **`GOOGLE_MAPS_API_KEY` 在新開的 worktree 裡可能是空的**：金鑰存在主目錄 `~/Projects/2026_nztrip/.env.local`，`.env.local` 不會被 git 追蹤，每個 worktree 要各自維護，開新 worktree 記得檢查/複製這個檔案。
- 排查照片問題不要只看畫面「這張看起來沒事」——瀏覽器 30 天快取會讓舊圖繼續顯示，掩蓋後端其實已經在回錯誤，要直接 curl 後端 API 確認。
- 同一天多列 Sheet 資料（例如不同家庭分開住宿）要先依「誰」分組再合併，不能直接攤平串成一個字串，會把不同家庭的資料混在一起。
- Google Places API（New）**沒有「封面照片」欄位**，抓到的照片不保證是最具代表性那張，這是 API 本身限制沒有可靠程式解法；私人 Airbnb 地址通常查不到照片也是正常現象，只有公共住宿設施查得到。
- **照片快取不會自己失效**：`/api/place-photo` 用查詢文字當 key，抓過就存伺服器磁碟＋DB（`place_photos`，**永久、無 TTL**）和瀏覽器 30 天。**只要查詢文字沒變就永遠回舊圖**。景點卡片曾只用「地名」當 key，Sheet 改了地址但地名沒變時照片不換，已改成「地名＋地址」組合 key（`Itinerary.tsx`），任一改動都會換新查詢→抓新圖。住房卡片本來就用地址當 key。若之後要「更新後立即全部重抓」，可加清 `place_photos` + `.photo-cache` 的機制。
- 部署到 Zeabur 的完整踩坑記錄（Vite 被誤判成靜態網站、Docker 架構不合、attestation 格式不相容）見 `docs/Zeabur部署教學.md`，不要重新踩一次。
- **iPhone 下載照片：`<a download>` 是元兇**。iOS Safari 只要看到連結有 `download` 屬性就把它當「下載檔案」丟給下載管理員（存進「檔案」App、跳一張灰色檔案卡），不會 inline 顯示圖片，使用者也就無法長按存到「照片」。後端回 `Content-Disposition: inline` 是對的但會被前端 `download` 蓋過。解法：用 `src/lib/utils.ts` 的 `isIOS` 分流——iPhone 不加 `download`、改 `target="_blank"` inline 開圖讓使用者長按存相簿（按鈕文案「開圖存相簿」）；桌機才用 `download` 一鍵下載。三個下載鍵在 `PhotoGallery.tsx`／`PhotoLightbox.tsx`。（注意：fetch→blob→download 在 iOS 一樣走下載管理員、進不了「照片」，別用。）
- **推新 image 後怎麼觸發 Zeabur 重新拉取**：實測「設定→來源」重存相同的映像位址/標籤 Zeabur 當作沒變、不會重新部署；改用服務狀態頁的「**重啟目前版本**」才會重新拉 `latest` 映像（對 `latest` 標籤有效）。驗證方式：`curl` 首頁抓 `assets/index-*.js` 檔名，跟本機 build 產出的檔名比對，或直接 grep 新文案。注意別把整包 minified JS 存進 shell 變數再 echo（會被截斷造成誤判），要 `curl ... | grep` 直接接。

## 環境/驗證備忘
- 啟動：`cd ~/Projects/2026_nztrip/.claude/worktrees/daily-summary && npm run dev`（port 3000）。lint：`npm run lint`。
- 這個專案的 `tsconfig` 不是 strict 模式，改介面加必填欄位時 `tsc --noEmit` 不會抓到 fallback 假資料忘記補欄位，要手動確認。
- **Claude in Chrome 分頁不在前景時，連 `window.scrollTo`／`el.scrollIntoView` 都不會生效**（不只是動畫凍結），這類行為沒辦法在這個環境裡端對端驗證，只能確認邏輯正確、請使用者實機測試。
- Claude in Chrome 視窗縮放無法真正模擬手機寬度，手機版排版請用瀏覽器 DevTools 裝置模式或真機確認。
- 每次改完務必 `npm run lint` 再 commit；部署到正式站要跑上面「部署資訊」那段指令，不是單純 `git push`。
